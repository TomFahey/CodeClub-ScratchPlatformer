Lesson 2

Levels & Scene Management

Transform the single-screen prototype into a multi-scene world — switching from colour-based collisions to a faster sprite-based system, refactoring the game loop into tidy named broadcasts, and building smooth scene transitions with a clever spawn-collision fix.

📋 What We'll Build

  • A dedicated Level sprite with multiple scene costumes
  • Faster, more flexible sprite-touching collision detection
  • A Check Touching Solid custom block to centralise all collision logic
  • A broadcast-driven game loop (Game Loop, Tick - Player, Tick - Last)
  • Tidy custom blocks for controls: Controls - Up and Down and Controls - Left and Right
  • A SCENE # variable that drives costume switching and scene transitions
  • Edge-of-screen detection so the player walks seamlessly between scenes
  • A Fix Collision in Direction (0) "wiggle" algorithm that prevents the player spawning inside the ground

1 Replacing the Backdrop with a Level Sprite

In Lesson 1 we drew our level directly onto the Stage backdrop and used a <touching color (#9966ff)?> block to detect collisions. That works fine for one colour, but as soon as we want colourful levels we'd need a separate colour check for every shade — and each extra check slows the game down. The fix is to move the level drawing into its own sprite so we can use a single, fast <touching (Level v)?> block instead.

Why? Sprite-touching is significantly faster in Scratch than colour-touching, and it detects collisions no matter how many colours are used in the costume. This means we can make beautiful, colourful levels without writing extra collision code or slowing the game.

  1. Click the Choose a Sprite button and create a brand-new, blank sprite. Name it Level.

  2. Click the Stage (bottom-right of the sprite list), then open the Backdrops tab. Click on your level backdrop costume so it is selected, then drag it across into the Level sprite's costume panel. This moves the level artwork into the new sprite.

  3. Back in the Stage's Backdrops tab, make sure nothing is selected, then click Delete to remove the old level backdrop, leaving the stage blank.

  4. Click the Level sprite and open its Costumes tab. If there is an empty placeholder costume at the top, select it and delete it so only your level artwork remains.

  5. Open the Code tab for the Level sprite and build this short startup script. Because the Level is now a sprite (not a backdrop), Scratch won't automatically centre it — so we position it at (0, 0) and push it behind all other sprites: 

    when gf clicked
go to x: (0) y: (0)
go to [back v] layer
    Why? Sprites render in front of the backdrop by default. Without go to [back v] layer, the Level sprite would draw on top of the player, hiding them completely!

  6. Now click the Player (Sprite1) sprite and find the define Move - in steps (steps) script. Replace the two <touching color (#9966ff)?> blocks — one in the horizontal check and one in the vertical check — with <touching (Level v)?>.

    Tip! The <touching (Level v)?> block is in the 🩵 Sensing category. Click the dropdown inside the block to choose the Level sprite from the list.
🎮 Try it! Click the green flag. The level should look exactly the same as before — but now the player will collide with every colour in the level, not just the original purple. Try drawing a few extra coloured shapes in the Level costume and confirm the player lands on them!

2 Centralising Collision Detection

Right now, every place in the code that needs to check for a collision contains its own <touching (Level v)?> block.As the game grows (moving platforms, lava, etc.) we'd need to update every single one of those blocks. Instead, let's create one custom block that handles all collision logic — so there's only one place to update in the future.

  1. Make sure the Player (Sprite1) sprite is selected. In the 🟥 Variables category, click Make a Variable and name it touching. Choose For this sprite only.

  2. In the 🟠 My Blocks (Custom Blocks) category, click Make a Block. Name it Check Touching Solid and tick the Run without screen refresh checkbox. Click OK.

    Why "Run without screen refresh"? This makes the custom block execute instantly within the same game tick, rather than waiting for the next screen update. Because we call this block inside the movement loop (which runs many times per frame), it must be fast — otherwise the game will lag noticeably.

  3. Build the define Check Touching Solid script. It simply checks whether the player is touching the Level sprite and stores the result in the touching variable: 

    define Check Touching Solid
if <touching (Level v)?> then
  set [touching v] to (1)
else
  set [touching v] to (0)
end
    Why a variable instead of using the block directly in the if? Scratch doesn't (yet!) let you drop a custom block directly into a condition — only built-in reporter blocks can go there. Storing the result in a variable like touching is the standard workaround, and it has a bonus: later we can add more collision checks (e.g. <touching (Lava v)?>) in this one block and the rest of the code automatically benefits.

  4. In the define Move - in steps (steps) script, update both collision checks. Before each if <touching (Level v)?> then, add a Check Touching Solid block and change the condition to (touching) > 0: 

    define Move - in steps (steps)
change [falling v] by (1)
repeat (steps)
  set [last_value v] to (x position)
  change x by ((speed_x) / (steps))
  Check Touching Solid
  if <(touching) > (0)> then
    set x to (last_value)
    set [speed_x v] to (0)
  end
  set [last_value v] to (y position)
  change y by ((speed y) / (steps))
  Check Touching Solid
  if <(touching) > (0)> then
    set y to (last_value)
    if <(speed y) < (0)> then
      set [falling v] to (0)
    end
    set [speed y v] to (0)
  end
end
🎮 Try it! Click the green flag and play as normal. The movement and collisions should feel identical to before — but the code is now much easier to extend later.

3 Organising the Code: Broadcasts and Custom Blocks

Our when gf clicked script is getting long — it currently handles initialisation, the game loop, jump controls, left/right controls, and movement all in one place. Before adding multiple scenes we're going to break it apart into clearly named pieces. This is a really good coding habit: small, named blocks are much easier to find and fix later on.

Why broadcasts instead of just one forever loop? A broadcast lets multiple sprites respond to the same event. Right now we only have one player sprite, but soon we'll add platforms, enemies, and UI elements — each of which can have its own when I receive [Tick - Player v] script. A single forever loop can only run code in one place; broadcasts scale to a whole game engine.

Step A — Split the game loop into a broadcast

  1. In the Player (Sprite1) sprite, find the when gf clicked script. Remove the forever loop and all the blocks inside it from this script.We're going to move those contents to a separate event.

  2. Create a new when I receive [Game Loop v] hat block (use the 🟡 Events category, choose broadcast and add a new message called Game Loop). Inside a forever loop, broadcast two new messages: 

    when I receive [Game Loop v]
forever
  broadcast [Tick - Player v]
  broadcast [Tick - Last v]
end
    Why "Tick"? A "tick" is one single frame of the game — like the tick of a clock. At 30 frames per second, 30 ticks happen every second. Naming broadcasts this way makes it immediately obvious they are frame-by-frame game updates.
    Why "Tick - Last"? Some logic (like checking whether the player walked off the edge of the screen) must happen after all movement has been applied. By putting it in a second broadcast that always runs after Tick - Player, we guarantee the correct order — every single frame.

  3. Create a when I receive [Tick - Player v] script. This is where all the per-frame player logic will live. For now it calls three blocks — we'll define those next: 

    when I receive [Tick - Player v]
Controls - Up and Down
Controls - Left and Right
Move - in steps (([abs v] of (speed y)) + ([abs v] of (speed_x)))

Step B — Create the Controls custom blocks

  1. Make a new custom block called Controls - Up and Down. Tick Run without screen refresh.Move the jump check and gravity into it: 

    define Controls - Up and Down
if <key [up arrow v] pressed?> then
  if <(falling) < (3)> then
    set [speed y v] to (JUMP FORCE)
  end
end
change [speed y v] by (GRAVITY)

  2. Make another custom block called Controls - Left and Right, also with Run without screen refresh.Move the arrow key checks and friction into it: 

    define Controls - Left and Right
if <key [left arrow v] pressed?> then
  change [speed_x v] by ((0) - (ACCELERATION))
end
if <key [right arrow v] pressed?> then
  change [speed_x v] by (ACCELERATION)
end
set [speed_x v] to ((speed_x) * (RESISTANCE))

Step C — Create the Reset and Begin Level custom block

  1. Make a new custom block called Reset and Begin Level. Important: do NOT tick "Run without screen refresh" this time — this block contains broadcasts that need to happen across multiple frames.

    Why not "run without screen refresh" here? Setting up a scene involves broadcasts that need to let other sprites respond (like the Level sprite switching costumes). If we ran without screen refresh, those other scripts would never get a chance to execute before we carry on.

  2. Build the define Reset and Begin Level script. It resets all player state and kicks off the game: 

    define Reset and Begin Level
set [SCENE # v] to (1)
set [speed y v] to (0)
set [speed_x v] to (0)
set [falling v] to (99)
go to x: (-174) y: (-55)
point in direction (90)
broadcast [Change Scene v]
broadcast [Game Loop v]
    Why set falling to 99? The jump check only allows jumping when falling is less than 3, which means the player is on the ground. Setting it to 99 at the start prevents the player from jumping in mid-air before they have even landed — they must touch the ground first.

  3. Update the when gf clicked script so it only sets the physics constants and then calls Reset and Begin Level. Also update GRAVITY to -1.5 and ACCELERATION to 2 for snappier movement: 

    when gf clicked
set [GRAVITY v] to (-1.5)
set [JUMP FORCE v] to (12)
set [ACCELERATION v] to (2)
set [RESISTANCE v] to (0.8)
Reset and Begin Level
🎮 Try it! Click the green flag. The game should feel exactly the same as before — but notice how much cleaner and easier to read the code is now. Movement might feel slightly faster because we increased ACCELERATION to 2.

4 Designing Multiple Scenes

Now for the exciting part — giving our world more than one screen to explore! We'll add extra costumes to the Level sprite (one per scene) and introduce a SCENE # variable so the game knows which scene to show.

  1. Click the Level sprite and open its Costumes tab. Rename the existing costume to exactly Scene1 (capital S, no space before the number — this is important!).

  2. Right-click on Scene1 in the costume list and choose Duplicate. Name the new costume Scene2.

  3. Design Scene2 however you like! A quick trick is to flip the scene horizontally — this automatically makes the right-hand side of Scene2 match the left-hand side of Scene1, so the player's floor never has a sudden height change at the border.

    Tip — The Centering Trick: Scratch flips around the costume's centre point, which can leave things off-centre. To flip around the canvas centre: select the Rectangle tool, draw the biggest rectangle that fits the canvas, then drag it so it snaps to the exact middle of the canvas. Now use the Selection tool to select everything, and click Flip Horizontal. Finally, delete the big rectangle. Everything will now be perfectly centred!

  4. Create a third costume called Scene3 (duplicate Scene2 and modify it). The costume name is automatically set to Scene3 when you duplicate, which is handy!

  5. Now let's make the Level sprite respond to scene changes. In the Level sprite's Code tab, add a new script using when I receive [Change Scene v]. Use a (join [Scene] (SCENE #)) block from 🟢 Operators to combine the word "Scene" with the SCENE # variable, then switch costume to that result: 

    when I receive [Change Scene v]
switch costume to (join [Scene] (SCENE #))
    Why join "Scene" with SCENE #? It constructs the costume name dynamically. When SCENE # is 1, this becomes "Scene1"; when it's 2, it becomes "Scene2", and so on. This means we never need to change this code even if we add 10 more scenes — just add more costumes with the right names!
🎮 Try it! In the Scratch variables display, manually click on SCENE # and change it to 2, then broadcast Change Scene by clicking the block. The level background should switch to Scene2. Change it back to 1 to return to the start.

5 Walking Off the Edge of the Screen

We have multiple scenes, but the player can't get to them yet! Let's detect when the player walks off the left or right edge and teleport them to the opposite side of the next (or previous) scene.

  1. In the Player (Sprite1) sprite, find the when I receive [Game Loop v] script. After the broadcast [Tick - Player v] line, add a new broadcast: broadcast [Tick - Last v].

  2. Create a new when I receive [Tick - Last v] script. Add an if <> then block that checks whether the player has gone off the right edge. The stage is 480 pixels wide (−240 to +240), so we use 235 as a safe threshold a few pixels inside the edge: 

    when I receive [Tick - Last v]
if <(x position) > (235)> then
  change [SCENE # v] by (1)
  broadcast [Change Scene v]
  set x to (-235)
end
if <(x position) < (-235)> then
  change [SCENE # v] by (-1)
  broadcast [Change Scene v]
  set x to (235)
end
    Why check in "Tick - Last" and not "Tick - Player"? Movement is calculated in Tick - Player. We need to check the final position after all that movement is done before deciding to change scene — otherwise the scene might switch halfway through a movement calculation, causing glitchy behaviour. Tick - Last always runs after Tick - Player, so the position is fully settled when we check it.
🎮 Try it! Click the green flag and walk the player all the way to the right edge. Do they appear on the left side of Scene2? Now walk back — do they return to Scene1? Try walking further right to reach Scene3!

6 Wrapping Scene Transitions into a Reusable Block

The left-edge and right-edge transition code is nearly identical — and we'll soon be adding more steps to it (the collision fix). Rather than duplicating everything, let's wrap it in a custom block called Begin Scene # (scene #) go to x: (x) so we only have to write (and maintain) the transition logic once.

  1. In the Player sprite's My Blocks, create a new custom block. Name it: Begin Scene # (scene #) go to x: (x). Add two number inputs: one called scene # and one called x. Do NOT tick "Run without screen refresh".

  2. Build the define script. Notice we set x to (x) to the target position before broadcasting Change Scene, so the player is already at the right place when the new scene appears. Then we stop the current game loop, wait a moment for the scene to load, and restart: 

    define Begin Scene # (scene #) go to x: (x)
set [SCENE # v] to (scene #)
set x to (x)
broadcast [Change Scene v]
stop [other scripts in sprite v]
wait (0) seconds
Fix Collision in Direction (0)
broadcast [Game Loop v] and wait
    Why "stop other scripts in sprite"? The game loop is still running when the scene transition begins. If we don't stop it, it will carry on moving the player and checking collisions on the old scene while we're trying to set up the new one, causing all sorts of strange bugs. Stopping it first means we can safely do the setup, then restart cleanly with a fresh broadcast [Game Loop v].
    Why "wait (0) seconds"? When we broadcast [Change Scene v] without "and wait", the costume switch is queued rather than happening immediately. A wait (0) seconds pauses just long enough for Scratch to process that queued event, so by the time we run the collision fix the Level costume has actually changed.

  3. Update the when I receive [Tick - Last v] script to use the new block instead of the inline code. Pass in the full calculated scene number and the target x position: 

    when I receive [Tick - Last v]
if <(x position) > (235)> then
  Begin Scene # ((SCENE #) + (1)) go to x: (-235)
end
if <(x position) < (-235)> then
  Begin Scene # ((SCENE #) - (1)) go to x: (235)
end
🎮 Try it! Scene transitions should work just as before — but now we have a single, clean block to maintain. You'll also notice a placeholder for the collision fix we're about to build in the next section!

7 Fixing Spawn Collisions: The Wiggle Algorithm

Here's a problem you'll run into: when the player moves to a new scene, they might appear inside the ground or a wall. This happens because the floor height at the right edge of Scene 1 might not perfectly match the left edge of Scene 2 — and in Scratch, "pixel perfect" is never guaranteed. If the player spawns inside solid ground, the collision system traps them there permanently. We need to automatically nudge them to safety.

Why does this happen? When we set x to (235) (reappearing on the right side of the previous scene), the player is placed at a fixed x coordinate but at whatever y coordinate they had before — which might be below the new scene's floor level. The collision detection then sees them inside the solid and refuses to let them move in any direction, locking them in place.

  1. In the Player sprite, create two new variables (both For this sprite only): temp and distance.

  2. Make a new custom block called Fix Collision in Direction (0) — add a number input named dir. This time, DO tick Run without screen refresh — it needs to run fast.

  3. Build the define script. This is the "wiggle" algorithm — it points the player in a direction (upward to start) and moves them a little further each loop until they're no longer stuck: 

    define Fix Collision in Direction (dir)
set [temp v] to (direction)
set [distance v] to (1)
point in direction (dir)
repeat (64)
  Check Touching Solid
  if <(touching) < (1)> then
    point in direction (temp)
    stop [this script v]
  end
  move (distance) steps
  turn right (180) degrees
  change [distance v] by (1)
end

How the wiggle algorithm works

Let's walk through what's happening step by step:

  1. Save the current direction in temp so we can restore it at the end — we don't want the player facing the wrong way after being un-stuck.

  2. Start distance at 1. On the first attempt, we'll move just 1 pixel. Each failed attempt, distance increases by 1, so we search 1 pixel away, then 2, then 3, and so on — always finding the nearest free spot.

  3. Point toward dir (0 = up). We start by searching upward because the player usually spawns slightly into the floor, so moving up is the fastest way to freedom.

  4. Inside the loop: check if we're still touching the level. If touching < 1 (not touching!), we've found a free spot — restore direction and stop the script immediately.

  5. If still touching: move forward by distance steps (in the current direction), then turn right (180) degrees to face the opposite direction. Increase distance by 1 and loop again.

    Why turn 180° each time? This is the "wiggle" — we alternate between moving up and moving down, each time going a little further. After moving 1 pixel up, we try 2 pixels down, then 3 pixels up, then 4 pixels down... We're searching an ever-growing zone around the original position, guaranteeing we find the closest free space in either vertical direction.

  6. The repeat (64) limit means we'll give up after 64 attempts. This prevents an infinite loop that would freeze Scratch if the player somehow ends up deeply stuck in a wall. You can increase this to 128 if you want extra tolerance for tricky level designs.

Tip — Test it visually! In the Scratch editor, manually position the player sprite so it overlaps the Level sprite. Then click directly on the define Fix Collision in Direction (0) block header to run it. Because you ran it by clicking (not via a broadcast), the "run without screen refresh" setting is ignored and you'll see the player wiggling up and down in slow motion until it pops free. Very satisfying!
Note — Bad level design still causes problems. If you place a wall across the full height of the scene exit, a player jumping into the next scene might spawn inside that wall. The wiggle algorithm can handle small overlaps, but can't escape a solid wall that fills the entire entry point. The best solution is good level design — don't place solid blocks right at the scene border.
🎮 Try it! Click the green flag. Walk off the right edge into Scene 2, then immediately walk back into Scene 1. Even if the floor heights differ slightly between scenes, the player should now pop smoothly onto the surface instead of getting stuck. Try jumping into the transition too!

✅ What We've Built

In this lesson we've transformed our single-screen prototype into the foundations of a real game engine. We moved from slow colour-based collisions to faster sprite-based detection, centralised all collision logic into a single Check Touching Solid block, and refactored the code into a clean broadcast-driven game loop with clearly named custom blocks. Most excitingly, we gave the game a whole world to explore — players can now walk between multiple scenes, with automatic collision fixing so they never get stuck in the ground.
