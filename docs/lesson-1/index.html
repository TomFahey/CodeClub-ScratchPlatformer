Lesson 1

Player Movement & Physics

Build the heart of a platformer game — a player sprite that falls with gravity, jumps, walks left and right, and reliably stops when it hits the ground.

📋 What We'll Build

  • A player sprite with a simple square costume and a basic level drawn in the backdrop
  • Realistic gravity and jumping using velocity variables and physics constants
  • Left and right movement with acceleration and friction (resistance)
  • Solid collision detection against the level colour using a clever "Move in steps" custom block
  • A "coyote jump" window that makes the controls feel fair and responsive

1 Setting Up the Project

Before we write any code, we need a player sprite with a costume and a backdrop that acts as our level. We're starting simple on purpose — a plain square player makes the physics much easier to get right before we add fancy graphics later.

  1. Open Scratch and start a new project. If you'd like to use griffpatch's starter project (which already contains all the Scratch Cat costumes you'll need in later lessons), look for the link under the original video and click Remix. Otherwise, a blank project is fine for now.

  2. Click on Sprite1 in the sprite panel at the bottom right. Rename it Player by clicking the name field at the top of the sprite panel and typing the new name.

    Why? Giving your sprite a meaningful name makes your project much easier to understand as it grows. "Player" tells you immediately what this sprite does.

  3. Click the Costumes tab at the top of the screen to open the costume editor. Click the Paint button (the brush icon) to create a new, blank costume.

  4. Select the Rectangle tool in the costume editor. Draw a small square to represent your player character — griffpatch calls his "Guy". The exact size doesn't matter, but keep it fairly small (around 30×30 pixels works well).

    Tip! After drawing your rectangle, drag it towards the centre crosshair of the drawing canvas until it snaps into place. It's really important that the costume is perfectly centred, otherwise the collision detection won't work correctly.

  5. Now let's draw the level. Click on the Stage thumbnail (bottom right, next to the sprite list), then click the Backdrops tab at the top of the screen.

  6. Using the Rectangle tool, draw a solid floor across the bottom of the stage. Pick a colour you like — griffpatch uses purple (#9966ff), and we'll be detecting this exact colour later. Make note of whichever colour you choose, as you'll need it again.

  7. Add a couple more rectangles to make some simple stairs and a floating platform. This gives us different shapes to test our collision detection against.

🎮 Try it! Click back on the Player sprite. You should see your square costume in the centre of the stage with your purple level behind it. Everything looking good? Great — let's start coding!

2 Adding Gravity

Gravity means the player accelerates downwards over time — they fall faster and faster the longer they're in the air. We'll use a variable called speed y to track how fast the player is currently moving vertically, and a constant called GRAVITY to control how quickly they speed up.

Make sure you're working on the Player sprite for this section.

  1. Click the Variables category (dark orange) in the blocks panel and click Make a Variable. Name it speed y and select For this sprite only, then click OK.

    Why "for this sprite only"? Vertical speed belongs to the Player — no other sprite needs to know about it. Keeping a variable local to its sprite avoids confusion as the project grows.

  2. Make a second variable. Name it GRAVITY and this time select For all sprites. We use capital letters to remind ourselves that this is a constant — a value we set once at the start and don't change during the game.

    Why a separate GRAVITY variable instead of just typing -1? Using a named variable means you can tweak the feel of the game in one place. If you decide gravity feels too floaty, just change the one set [GRAVITY ▾] block — you don't have to hunt through all your code.

  3. Click the Events category (yellow) and drag a when 🚩 clicked block into the scripts area.

  4. From Variables, attach a set [GRAVITY ▾] to (0) block and change the value to -1. Then attach a set [speed y ▾] to (0) block below it.

    Why set speed y to 0 at the start? Scratch remembers variable values between runs. By resetting everything when the green flag is clicked, we make sure the player always starts fresh — not carrying over speed from a previous run.

  5. From Motion (blue), attach a go to x: (-150) y: (55) block. This puts the player at a starting position just above the floor.

  6. From Control (orange), attach a forever loop block below the go-to block. Everything that happens every frame of the game will live inside this loop.

  7. Inside the forever loop, place a change [speed y ▾] by (GRAVITY) block (drag the GRAVITY variable into the number slot). Then directly below that, add a change y by (speed y) block.

    when gf clicked
set [GRAVITY v] to (-1)
set [speed y v] to (0)
go to x: (-150) y: (55)
forever
  change [speed y v] by (GRAVITY)
  change y by (speed y)
end
    Why change speed y by GRAVITY instead of just changing y by -1? If we moved the player down by a fixed amount each frame, they'd fall at a constant speed — like a slow elevator. Real gravity accelerates objects: the longer you fall, the faster you go. Each frame, GRAVITY (-1) is added to speed y, making speed y more and more negative, which makes the player fall faster and faster. That's acceleration!
🎮 Try it! Click the green flag. The player should fall off the screen — they accelerate as they go. Don't worry that they fall through the floor for now — that's exactly what we'll fix next!

3 Detecting the Ground

Now we'll make the player stop when they hit the level. We'll detect when the player overlaps the level colour and then nudge them back out. We'll wrap this nudging logic in a custom block that runs invisibly, so the player appears to land cleanly on the ground.

  1. Inside the forever loop, directly below the change y by (speed y) block, add an if <> then block from Control.

  2. From Sensing (light blue), drag a touching color [ ]? block into the condition slot of the if block. Click the coloured square in the block and use the colour picker (the eyedropper tool) to click on the purple level colour in the stage. The block's colour swatch should now match your level.

    Tip! Zoom into the stage a little before using the eyedropper — it's easier to pick exactly the right pixel of your level colour.

  3. Inside the if block, we need to push the player back up out of the ground. The safest way is to move them up one pixel at a time until they're no longer touching the level. From Control, place a repeat until <> block inside the if. In its condition, put a not block (from Operators, green) containing another touching color [ ]? block with the same level colour.

  4. Inside the repeat until loop, add a change y by (1) block. This nudges the player up by one pixel each pass until they're clear of the ground.

    Why move up one pixel at a time? If we simply moved back by the same distance we fell, we could overshoot — the player would float above the ground or bounce around. Moving just one pixel at a time guarantees we stop at precisely the surface, no matter how fast the player was falling.

  5. After the repeat until loop (but still inside the outer if block), add a set [speed y ▾] to (0) block. This resets the vertical speed to zero when the player lands — otherwise they'd immediately start falling again.

  6. Now let's make this nudging invisible. From the My Blocks category (pink), click Make a Block. Name it Fix Overlap. Crucially, tick the box that says Run without screen refresh, then click OK.

    Why "Run without screen refresh"? Without this option, Scratch would show the player visibly sliding up through the ground one pixel at a time each frame — it would look terrible! With this option ticked, the entire nudge-out loop finishes in the same frame before the screen updates, so the player appears to land cleanly and instantly.

  7. A new define Fix Overlap block appears in your scripts area. Drag the repeat until loop (from inside the if block) and attach it underneath the define block. Replace it in the if block with the new Fix Overlap block.

    forever
  change [speed y v] by (GRAVITY)
  change y by (speed y)
  if <touching color (#9966ff)?> then
    Fix Overlap
    set [speed y v] to (0)
  end
end

    define Fix Overlap
repeat until <not <touching color (#9966ff)?>>
  change y by (1)
end
🎮 Try it! Click the green flag. The player should now fall and land cleanly on the floor without sinking in. Try starting the player at different heights — they should always settle neatly on the surface.

4 Jumping and Horizontal Movement

Now for the fun part — making the player controllable! We'll add a jump, left and right movement, and importantly some friction so the player slows down naturally instead of sliding forever. We're still working on the Player sprite.

  1. At the very top of the forever loop (above the gravity change), add an if <key [up arrow ▾] pressed?> then block from Control and Sensing. Inside it, add a set [speed y ▾] to (12) block.

    Why set speed y to 12? Setting speed y to a large positive number gives the player an immediate upward burst of velocity. Gravity then gradually reduces this speed each frame until the player starts falling again — exactly like a real jump arc!

  2. Make a new variable. Name it speed_x and select For this sprite only. This will track how fast the player is moving left or right.

  3. Back in the setup code (at the top, under when 🚩 clicked), add a set [speed_x ▾] to (0) block alongside the other initialisations.

  4. Inside the forever loop, below the jump block, add an if <key [left arrow ▾] pressed?> then block. Inside it, add change [speed_x ▾] by (-1.5).

    Why change speed_x instead of set it? Changing (adding to) the speed each frame means the player accelerates gradually — they build up speed rather than jumping instantly to full speed. This feels much more natural and satisfying to control.

  5. Add another if <key [right arrow ▾] pressed?> then block below. Inside it, add change [speed_x ▾] by (1.5).

  6. Below the left/right key checks, add a set [speed_x ▾] to ((speed_x) * (0.8)) block. This multiplies speed_x by 0.8 every single frame, which gradually shrinks it towards zero — simulating friction.

    Why multiply by 0.8? Any value between 0 and 1 will cause the speed to shrink each frame. A value close to 1 (like 0.95) feels slippery; a value closer to 0 (like 0.5) stops the player very abruptly. 0.8 gives a nice responsive feel — the player slows down noticeably but not harshly.

  7. Below the resistance line and above the gravity line, add a change x by (speed_x) block to actually move the player horizontally each frame.

    forever
  if <key [up arrow v] pressed?> then
    set [speed y v] to (12)
  end
  if <key [left arrow v] pressed?> then
    change [speed_x v] by (-1.5)
  end
  if <key [right arrow v] pressed?> then
    change [speed_x v] by (1.5)
  end
  set [speed_x v] to ((speed_x) * (0.8))
  change x by (speed_x)
  change [speed y v] by (GRAVITY)
  change y by (speed y)
  if <touching color (#9966ff)?> then
    Fix Overlap
    set [speed y v] to (0)
  end
end
🎮 Try it! Click the green flag and use the arrow keys. The player should walk left and right (slowing down when you let go) and jump with the up arrow. Notice there's no left/right wall collision yet — the player walks straight through the stairs. We'll fix that in the next section!

5 Smooth Collision Detection: "Move in Steps"

Right now our collision system has a problem: the player can walk straight through walls, and at high speeds might sink deep into the ground before the collision check catches it. We're going to replace the old movement code with a much smarter technique: instead of moving the player a large distance and then backing them out, we'll move in many tiny steps and stop the moment a collision is detected.

This is the most important part of this lesson, so take your time with it!

  1. From My Blocks, click Make a Block. Name it Move - in steps. Click Add an input (number or text) and label it steps. Make sure Run without screen refresh is ticked. Click OK.

    Why "Run without screen refresh" again? This custom block will loop many times per frame doing tiny movements. If it refreshed the screen after every micro-step, the game would slow to a crawl. With this option on, all the micro-steps happen invisibly in the same frame, so the game runs at full speed.

  2. Make a new variable called last_value, for this sprite only. We'll use this to remember the player's position just before each micro-step, so we can undo the move if a collision happens.

  3. In the define Move - in steps (steps) block, add a repeat (steps) loop. All of the movement and collision code will go inside this loop.

  4. Inside the repeat loop, handle the horizontal (X) movement first:

    • Add set [last_value ▾] to (x position) — save where we are before moving.
    • Add change x by ((speed_x) / (steps)) — move a tiny fraction of the total distance.
    • Add an if <touching color [#9966ff]?> then block. Inside it, add set x to (last_value) and set [speed_x ▾] to (0).
    Why divide speed_x by steps? We want to travel the full distance of speed_x pixels total, but spread it over many tiny steps. If we take 10 steps, each step moves 1/10 of the total distance. That way the player ends up in exactly the right place — but we get to check for collisions after each tiny move.

  5. Directly below the X collision block (still inside the repeat loop), handle the vertical (Y) movement:

    • Add set [last_value ▾] to (y position) — save the Y position before moving.
    • Add change y by ((speed y) / (steps)) — move a tiny fraction vertically.
    • Add an if <touching color [#9966ff]?> then block. Inside it, add set y to (last_value) and set [speed y ▾] to (0).

  6. Here's what the complete define Move - in steps block looks like so far:

    define Move - in steps (steps)
repeat (steps)
  set [last_value v] to (x position)
  change x by ((speed_x) / (steps))
  if <touching color (#9966ff)?> then
    set x to (last_value)
    set [speed_x v] to (0)
  end
  set [last_value v] to (y position)
  change y by ((speed y) / (steps))
  if <touching color (#9966ff)?> then
    set y to (last_value)
    set [speed y v] to (0)
  end
end

  7. Now wire up the main loop to use this new block. In the forever loop, remove the change x by (speed_x) block, the change y by (speed y) block, and the old if touching color / Fix Overlap block (you can also delete the Fix Overlap definition — it's no longer needed). Replace them all with a single Move - in steps ( ) block.

  8. For the steps input, we need to pass in a number that's large enough to ensure we never move more than 1 pixel per micro-step. That number is the total distance we're about to travel. From Operators (green), build: (abs of (speed y)) + (abs of (speed_x)). The abs (absolute value) block is found inside the green ( ) of ( ) maths block — click its dropdown and choose abs. This converts negative speeds to positive numbers for the calculation.

    forever
  if <key [up arrow v] pressed?> then
    set [speed y v] to (12)
  end
  if <key [left arrow v] pressed?> then
    change [speed_x v] by (-1.5)
  end
  if <key [right arrow v] pressed?> then
    change [speed_x v] by (1.5)
  end
  set [speed_x v] to ((speed_x) * (0.8))
  change [speed y v] by (GRAVITY)
  Move - in steps (([abs v] of (speed y)) + ([abs v] of (speed_x)))
end
    Why use abs(speed y) + abs(speed_x)? This gives us the total number of pixels we're about to move in this frame (combining both directions). By running that many micro-steps, we guarantee each step is at most 1 pixel — which means a collision is always caught before we've moved more than 1 pixel into a wall or floor. No tunnelling!
🎮 Try it! Click the green flag and test the movement. Now try running into the stair steps — the player should stop and slide against them rather than passing through. Try bumping your head on the underside of the floating platform too. Solid collisions all round!

6 Fixing the Jump and Fine-Tuning

We have one last problem: hold the up arrow key and the player flies off into the sky like a rocket! We need to restrict jumping so it only works when the player is on the ground (or very close to it). We'll also fix a bug where the player can stick to ceilings, and then extract the remaining magic numbers into named constants so the game is easy to tweak.

  1. Make a new variable called falling, for this sprite only. This will count how many frames the player has been off the ground. When it's 0, the player is on the ground; when it's large, the player has been in the air a while.

  2. At the very top of the define Move - in steps (steps) block (before the repeat loop), add a change [falling ▾] by (1) block. Every frame, this increments the airborne counter.

  3. Inside the Y-axis collision check in Move - in steps (the if block that runs when the player hits a floor or ceiling), add a check before resetting speed y. Wrap a new if <(speed y) < (0)> then block around a set [falling ▾] to (0) block, and place this above the set [speed y ▾] to (0) line.

    Why check that speed y is negative (less than 0)? A negative speed y means the player was moving downwards when the collision happened — that means they hit a floor. A positive speed y means they were moving upwards and hit a ceiling. We only want to reset the falling counter for floor landings. Without this check, a player holding the jump key could stick to the ceiling! 
    define Move - in steps (steps)
change [falling v] by (1)
repeat (steps)
  set [last_value v] to (x position)
  change x by ((speed_x) / (steps))
  if <touching color (#9966ff)?> then
    set x to (last_value)
    set [speed_x v] to (0)
  end
  set [last_value v] to (y position)
  change y by ((speed y) / (steps))
  if <touching color (#9966ff)?> then
    set y to (last_value)
    if <(speed y) < (0)> then
      set [falling v] to (0)
    end
    set [speed y v] to (0)
  end
end

  4. Now go back to the main forever loop. Find the if <key [up arrow ▾] pressed?> then block. Wrap the set [speed y ▾] to (12) block inside a new inner if <(falling) < (3)> then block.

    Why < 3 instead of = 0? This gives the player a 2-frame window to jump after walking off the edge of a platform. Players often press jump slightly too late, and without this forgiveness they'd feel cheated. This trick is called coyote time — named after a certain cartoon coyote who keeps running briefly after going off a cliff! It makes the game feel fair without any cheating. 
    if <key [up arrow v] pressed?> then
  if <(falling) < (3)> then
    set [speed y v] to (12)
  end
end

  5. Now let's replace the remaining "magic numbers" with named constants. Make three new variables, all for all sprites: JUMP FORCE, ACCELERATION, and RESISTANCE.

  6. At the top of the when 🚩 clicked script, add these three set blocks alongside the GRAVITY initialisation:

    set [JUMP FORCE v] to (12)
set [ACCELERATION v] to (1.5)
set [RESISTANCE v] to (0.8)

  7. Replace the hard-coded numbers throughout the forever loop with the new variables:

    • Change set [speed y ▾] to (12) → set [speed y ▾] to (JUMP FORCE)
    • Change change [speed_x ▾] by (-1.5) → change [speed_x ▾] by ((0) - (ACCELERATION))
    • Change change [speed_x ▾] by (1.5) → change [speed_x ▾] by (ACCELERATION)
    • Change set [speed_x ▾] to ((speed_x) * (0.8)) → set [speed_x ▾] to ((speed_x) * (RESISTANCE))

  8. Here's the complete final version of the when 🚩 clicked script:

    when gf clicked
set [GRAVITY v] to (-1)
set [JUMP FORCE v] to (12)
set [ACCELERATION v] to (1.5)
set [RESISTANCE v] to (0.8)
set [speed_x v] to (0)
set [speed y v] to (0)
go to x: (-150) y: (55)
forever
  if <key [up arrow v] pressed?> then
    if <(falling) < (3)> then
      set [speed y v] to (JUMP FORCE)
    end
  end
  if <key [left arrow v] pressed?> then
    change [speed_x v] by ((0) - (ACCELERATION))
  end
  if <key [right arrow v] pressed?> then
    change [speed_x v] by (ACCELERATION)
  end
  set [speed_x v] to ((speed_x) * (RESISTANCE))
  change [speed y v] by (GRAVITY)
  Move - in steps (([abs v] of (speed y)) + ([abs v] of (speed_x)))
end
Tip! Try adjusting the constants! Change GRAVITY to -1.5 for a heavier feel, JUMP FORCE to 20 for a super jump, or RESISTANCE to 0.95 for an icy floor. Because all these values are in one place at the top of the script, experimenting is easy.
🎮 Try it! Click the green flag and put your platformer through its paces. Walk up the stairs, jump onto the floating platform, run into walls, and try jumping just after walking off a ledge. Everything should feel solid and responsive. Well done — you've built a working physics engine from scratch!

✅ What We've Built

In this lesson we built a complete physics-based player controller entirely from scratch. The player falls with realistic accelerating gravity, jumps cleanly, walks left and right with smooth acceleration and friction, and collides accurately against the level using the "Move in steps" sub-step technique — which prevents the player from ever tunnelling through a surface no matter how fast they're moving.

We also added a "coyote jump" window using the falling counter, so players can jump briefly after stepping off a ledge — a small touch that makes a big difference to how fair and fun the game feels.
